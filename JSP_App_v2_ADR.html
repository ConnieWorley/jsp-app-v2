001
Tech Stack: React + Vite, Supabase, Anthropic Claude API
Accepted
Context

JSP App v2 requires a modern frontend framework, a managed backend with auth and file storage, and integration with a large language model for AI-assisted features. The app is a solo build targeting personal use in v2, with a path toward multi-user SaaS in v3.

Decision

React + Vite for the frontend, Supabase for database, auth, and file storage, and the Anthropic Claude API for all AI features. Claude Code is used as the implementation tool for the Developer role.

Options Considered
React + Vite / Supabase / Anthropic API
Lightweight build tooling, industry-standard frontend framework, fully managed backend with built-in auth and storage, and direct access to Claude's capabilities. Strong documentation and community support. Scales to SaaS without re-platforming.
Selected
Next.js / Supabase / Anthropic API
Next.js adds server-side rendering and API routes, which are useful for protecting API keys — but introduces complexity not needed for a single-user v2 app. The added learning curve was not justified by the v2 requirements.
Not selected
Vue or Svelte alternatives
Both are viable but have smaller ecosystems for the specific combination of Supabase integration and Claude Code tooling. React was the stronger fit for the portfolio audience and the available AI development tooling.
Not selected
Consequences
▲Vite provides fast local development with hot module replacement — important for a solo builder moving quickly.
▲Supabase handles auth, database, and file storage in one platform — eliminates the need to manage multiple backend services.
▲Stack is well-supported by Claude Code, enabling AI-assisted implementation.
▼Auth hardening and multi-user row-level security are deferred to v3 — acceptable given the single-user v2 scope.
002
API Call Architecture: Client-Side Only, No Server Middleware
Accepted
Context

The app calls both Supabase and the Anthropic Claude API. A decision was required on where those calls originate — from the React client directly, or via a server-side layer such as Supabase Edge Functions.

Decision

All API calls originate client-side from React. No Edge Functions or server middleware are used in v2.

Options Considered
Client-side calls (React → Supabase + Claude API directly)
Simpler architecture, faster to build, no additional infrastructure to manage. Appropriate for a single-user app with no public exposure of the API key in a shared environment.
Selected
Server-side via Supabase Edge Functions
Keeps the Anthropic API key off the client — the stronger security posture for a public multi-user app. However, adds architectural complexity not justified for a single-user v2 build. Deferred to v3 when multi-user support and auth hardening are introduced.
Deferred to v3
Consequences
▲Significantly simpler implementation — no Edge Function deployment, no middleware layer to debug.
▲Faster path to a working app, which is the primary v2 goal.
▼Anthropic API key is stored client-side (environment variable). Acceptable for single-user personal use; must be addressed before v3 multi-user launch.
▼v3 will require refactoring API calls to move through a server layer — a known and planned cost.
003
JD Import: Paste-Only, No URL Fetch or Scrape
Accepted
Context

Job descriptions need to enter the app as text for scoring and analysis. The question was whether to attempt fetching JD content from a pasted URL, or to rely on user paste.

Decision

JD import is paste-only. Users may optionally paste the source URL, which is stored for reference only. No fetch or scrape is attempted.

Options Considered
URL paste + JD text paste (no fetch)
Clean, reliable UX. The URL is captured for reference and traceability without attempting a fetch that would frequently fail due to CORS restrictions on job board sites. A failing fetch signals a broken app to the user even when the failure is caused by external restrictions outside the app's control — a poor user experience that misrepresents the quality of the build.
Selected
Attempt URL fetch, fall back to paste on failure
Superficially more convenient, but most job board URLs (LinkedIn, Indeed, Workday, Greenhouse) block client-side fetches via CORS headers. Frequent failures create a negative first impression. Without a server-side proxy, reliable scraping is not achievable in a client-only architecture.
Not selected
Consequences
▲Reliable, consistent UX — no spinners, no silent failures, no user confusion.
▲URL is preserved for reference, traceability, and potential v3 server-side scraping.
▼Requires one extra user step (copy-paste JD text). Acceptable given the reliability tradeoff.
004
Checklist Model: Master Template + Per-Job Instance
Accepted
Context

Each job moves through three stages (application, networking, interview), each with its own checklist. A decision was required on whether checklists are configured per-job manually or instantiated from a master template.

Decision

One master checklist template per stage, configured once in Setup. When a job transitions to a new stage, the template is instantiated as a per-job copy. Users may edit individual instances without affecting the master template.

Options Considered
Master template + per-job instance
Configure once, reuse across every job. Per-job instances are editable without touching the template. Due dates are auto-calculated at instantiation from offsets set in Setup, with user override available. Scales cleanly as more jobs are added.
Selected
Manually assign checklist items per job
Maximum flexibility per job but creates significant repetitive data entry — the exact problem this app is designed to eliminate. Rejected as inconsistent with the app's core value proposition.
Not selected
Consequences
▲Minimal data entry per job — the template does the work once.
▲Auto-calculated due dates from Setup offsets reduce manual scheduling entirely.
▲Per-job editability preserves flexibility for exceptions without corrupting the template.
▼Requires two tables (checklist_templates + checklist_instances) — slightly more complex data model, but a clean and well-understood pattern.
005
Data Model: PostgreSQL Enums for Single-Option Fields, text[] for Multi-Select
Accepted
Context

Several fields in the data model have a defined, bounded set of valid values (e.g., job status, gap type, outreach type). A decision was required on how to enforce those constraints at the database level and how to present them in the UI.

Decision

Single-option fields with defined values are implemented as PostgreSQL enums, rendered as dropdowns or radio buttons in the UI. Multi-select fields are implemented as text[] (PostgreSQL arrays), rendered as checkbox groups. Free text entry is minimized throughout.

Rationale

Enums enforce data integrity at the database level, prevent invalid entries, and make the UI predictable. Minimizing free text entry reduces cognitive load, speeds up data entry, and produces cleaner, more queryable data — all consistent with the app's goal of reducing friction in the job search process. The one explicit multi-select field is networking.outreach_type, where a contact may have been reached via multiple channels simultaneously.

Consequences
▲Data integrity enforced at the database level — no invalid status values possible.
▲UI is faster and more consistent — users select rather than type.
▲Cleaner data supports reliable filtering, sorting, and dashboard aggregation.
▼Adding new enum values requires a database migration — a minor operational cost if the value set needs to expand.
006
Target Companies: Independent Entity, Not Attached to Jobs
Accepted
Context

The Ideal Company Quiz produces a profile of companies the user wants to work for. A decision was required on whether that company data lives inside the job record or as a standalone entity.

Decision

target_companies is a standalone table, independent of any specific job. Jobs may optionally link to a target company via a nullable foreign key, but a company record can exist without any associated job.

Rationale

A target company is a strategic career preference, not a job application artifact. The user may be tracking a company they want to work for before any specific role opens up. Treating it as independent preserves that intent and allows the ideal company fit score to be calculated at the company level and referenced across multiple job applications at the same company.

Consequences
▲Company preferences survive independently of individual job applications.
▲Multiple jobs at the same company can reference the same company record and ideal fit score.
▲Supports proactive company tracking before a specific role is identified.
▼Requires the user to link a job to a target company manually if desired — a minor UX step, not a workflow burden.
